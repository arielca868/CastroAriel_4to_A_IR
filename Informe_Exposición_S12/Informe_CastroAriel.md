# Exposición: Herramientas de Desarrollo Dirigido por Modelos (MDD)

## Grupo H: Enterprise Architect

### Expositor: Huilcapi Fabiola

**Teoría**
- Definición y propósito de Enterprise Architect.
- Es una plataforma de modelado orientada tanto al ámbito académico como al empresarial.
- Estándares compatibles: UML 2.5, BPMN y SysML.

**Práctica**
- Elaboración de un diagrama de clases.
- Incorporación de atributos.
- Creación de operaciones.
- Definición de relaciones entre clases.

### Expositor: Vaca

**Teoría**
- El modelado se maneja de forma integral, cubriendo todo el ciclo.
- Permite mantener trazabilidad completa entre los elementos del proyecto.
- Cuenta con un repositorio central donde se almacena la información.
- Tipos de diagramas disponibles: casos de uso, clases, estados, actividades y secuencia.
- Facilita la gestión, documentación e ingeniería inversa.

**Práctica**
- Pasos para crear un nuevo proyecto.
- Creación de un diagrama.
- Revisión de la estructura general.
- Elementos que componen una clase.
- Creación de una clase con sus atributos.

### Expositor: Tigasi

**Teoría**
- Puntos a favor: herramienta muy completa e integradora, buen desempeño en proyectos grandes, precio competitivo y buena reputación en el mercado.
- Puntos en contra: requiere tiempo de aprendizaje, tiene costo, y funciona como programa de escritorio.
- Cuenta con reconocimiento tanto académico como profesional.

**Práctica**
- Proceso de generación de código.
- Elección del lenguaje de programación.
- Generación del código final.
- Aplicación de ingeniería inversa (de código a diagrama).

## Grupo A: StarUML

### Expositor: Steven Díaz

**Teoría**
- ¿En qué consiste el desarrollo dirigido por modelos (MDD)?
- Conceptos relacionados: MDE, MDA, MDD, M2M y M2T.
- Es una herramienta CASE pensada para el modelado en UML.
- Admite extensiones para poder generar código automáticamente.

**Práctica**
- Apertura de un proyecto.
- Procedimiento para generar código.
- Instalación de una extensión para generación de código.

### Expositor: Escudero

**Teoría**
- Es compatible con la versión UML 2.5.
- Puede generar código en C# de forma automática.
- Su interfaz es simple y de fácil manejo.
- Cuenta con un subsistema dedicado al modelado.
- Explicación de sus procesos principales.

**Práctica**
- Modificación de un diagrama existente.
- Aplicación de ingeniería inversa.
- Conversión de código a diagrama.

### Expositor: Erick Mera

**Teoría**
- Trabajo centrado en el modelo UML.
- Beneficios: funciona en varias plataformas, es ligero y rápido, y es de código abierto.
- Tiene una interfaz consolidada, cumple con el estándar y permite exportar de forma flexible.
- Proceso de transformación de UML a código C#.

**Práctica**
- Manejo del entorno visual.
- Apertura de una clase o código generado.
- Importación de clases.
- Vinculación con el IDE de Visual Studio.

### Expositor: Mesías Jhon

**Teoría**
- Puntos débiles: la curva de aprendizaje y la falta de integración con otras herramientas.
- Nivel de compatibilidad de la herramienta.
- Comparación de funciones, generación de código y manejo de archivos del proyecto.
- Qué tan fácil es de usar frente a Visual Paradigm.

**Práctica**
- Generación de código.
- Elección de la carpeta destino para el código generado.

## Grupo E: Eclipse Papyrus + Acceleo

### Expositor: Gamarra

**Teoría**
- Qué es el MDD: diseño, modelo, reglas y transformación.
- Diferencias entre MDE, MDA y MDD.
- Alcance del sistema y su nivel de formalización.
- Es una herramienta para modelar en UML.
- Cumple con los estándares definidos por la OMG.
- Ventajas: es software libre y permite extensiones.
- Uso de UML y plantillas, entre otros elementos.

**Práctica**
- Creación de un modelo.
- Configuración del entorno combinando Acceleo y Papyrus.
- Ubicación de la carpeta donde se guarda el código.
- Demostración práctica de cómo el código se refleja en el diagrama.

### Expositor: Pérez

**Teoría**
- Explicación de los niveles de abstracción.
- Tipos de transformación en MDD: M2M y M2T.
- Beneficios de este enfoque.
- Cómo se adapta según el lenguaje.
- El PIM permite diseñar el sistema sin depender de una plataforma específica.
- La mayor parte del código se genera automáticamente desde el modelo.

**Práctica**
- Generación de código.
- Exportación de una plantilla.
- Configuración de la ejecución.
